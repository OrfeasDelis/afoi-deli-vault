# Session 2026-06-17 — Materials layer goes three-tier (class → collection → SKU)

## Context loaded
- Continues directly from [[Session 2026-06-16d]] (materials Batch 1 + the repeatable [[Materials Research Workflow]], paused for steer).
- Orfeas raised the core architecture question: the generic material note / [[Materials Schema]] can't hold the *specifics* of an actual line — Kronos **Pierre Vive** ≠ Florim **Authentic Luxe**; Inda furniture ≠ Antonio Lupi furniture; Laufen SaphirKeramik ≠ Cielo VC. Dimensions, thicknesses, surface wording differ and are often *far more* than the class note carries. "Schemas/notes per supplier, or keep as-is?"

## Work done — designed (via brainstorming) and encoded a three-tier model

**Decision (brainstormed, approved by Orfeas) → [[Architecture Decision Records|ADR-0005]]:**
- **Three tiers, one shared vocabulary:** `material` (class · ~22 · controlled vocab · [[Materials Schema]]) → `collection` (instance · hundreds · [[Collection Schema]]) → `SKU` (orderable · OS Postgres `products`).
- **Not** a schema per supplier (that destroys the shared queryable vocabulary). One shared Collection schema; the differences live in *values* + a **raw + normalized** capture (brand's literal "Bocciardato" **and** `finishes: [bush-hammered]`) — this dissolves the cross-brand finish/format-wording problem.
- **Collections are born on ingestion, never pre-stubbed** (Orfeas chose Option A). Created when a source is fed or a project uses the line; until then the line is a roster row (✅ ingested / ⬚ not-yet) in `<Supplier> — Collections Reference`. Kills stub-sprawl while keeping every collection a first-class, linkable entity (projects/quotes link the exact line).
- **Honest cost noted:** file-count + graph density grow over time; requires discipline not to pre-stub.

**Ingestion mechanism (Orfeas asked to specify it):** a collection note is built from whichever real source exists — (1) manufacturer URL, (2) **a fed/attached PDF or TDS** (primary high-fidelity — read via the Read tool, archived under the supplier's `_sources/`, listed in `source_files`), (3) mixed, or (4) **future/uncertain: a Google Drive feeder** (Orfeas uploads PDFs; Claude or the worker ingests on prompt or on a watch — depends on the worker's Drive access, the same parked service-account-vs-OAuth decision). Modes 1–3 are live now.
